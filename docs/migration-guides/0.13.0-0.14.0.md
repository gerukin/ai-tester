# Migration guide: v0.13.0 to v0.14.0

This guide is for upgrading an existing project from `v0.13.0` to `v0.14.0`.

The goal is to make the upgrade a no-op from a testing history point of view:

- existing sessions and evaluations stay attached to the same logical models
- no old models disappear unexpectedly
- running missing tests or missing evaluations after the migration should find nothing new to do unless you intentionally changed your config, prompts, tests, or model definitions

## What changed

This release moves provider and model registration to YAML files:

- providers now come from `AI_TESTER_PROVIDERS_DIR`
- models now come from `AI_TESTER_MODELS_DIR`
- model costs come from each model YAML file
- per-model runtime options such as provider-specific request fields and thinking settings are now part of the model version definition

The main config file still references models with:

```yaml
provider: openai
model: gpt-4o-mini-2024-07-18
```

The `model` value is still the provider-facing `providerModelCode`.

## Before you start

1. Read the [changelog](/CHANGELOG.md) entry for `v0.14.0`.
2. Make a backup of your SQLite database file.
3. Make sure your current project is not mid-run.
4. Make sure your current tests/evals are already in a good state before upgrading.

If you are unsure where your DB is, check `AI_TESTER_SQLITE_DB_PATH` in your env file.

## 1. Update the package

Update the package version first:

```sh
npm update @gerukin/ai-tester
```

If you use the repository directly rather than the published package, update your checkout to the target version and run:

```sh
npm install
npm run compile
```

## 2. Add the required registry env vars

Your env file must define:

```env
AI_TESTER_PROVIDERS_DIR=.local/ai-evals-data/providers
AI_TESTER_MODELS_DIR=.local/ai-evals-data/models
```

Use whatever paths are appropriate for your project, but both directories must exist before running migrations or the app.

## 3. Create the provider and model directories

Create both directories on disk:

```sh
mkdir -p .local/ai-evals-data/providers
mkdir -p .local/ai-evals-data/models
```

## 4. Convert providers to provider-only YAML files

Create one provider YAML file per provider under `AI_TESTER_PROVIDERS_DIR`.

Provider files must contain only provider registration data. Do not keep nested `models:` arrays in provider files.

Example:

```yaml
code: openrouter
name: OpenRouter
type: openai-compatible
baseURL: https://openrouter.ai/api/v1
apiKeyEnvVar: OPENROUTER_API_KEY
```

## 5. Convert each model version into its own YAML file

Create one model YAML file per runnable provider model under `AI_TESTER_MODELS_DIR`.

Each model YAML must include:

- `code`
- `provider`
- `providerModelCode`
- optional `extraIdentifier`
- optional `active`
- full `costs` history

If you previously relied on built-in runtime behavior for a model, backport that behavior into YAML explicitly now.

Examples:

- Ollama reasoning extraction:

```yaml
thinking:
  extractionTagName: think
```

- OpenAI or OpenAI-compatible reasoning effort:

```yaml
thinking:
  effort: low
```

- Vertex Gemini 2.5 thinking settings:

```yaml
thinking:
  includeThoughts: true
candidateOverrides:
  thinking:
    budgetTokens: 5000
evaluatorOverrides:
  thinking:
    budgetTokens: 2000
```

- Anthropic-style thinking settings:

```yaml
thinking:
  enabled: true
candidateOverrides:
  thinking:
    budgetTokens: 5000
evaluatorOverrides:
  thinking:
    budgetTokens: 2000
```

## 6. Make sure cost history is complete

For every active model version with historical runs:

- the `costs` array in YAML must cover the oldest existing session or evaluation for that model version
- stale DB cost rows will be removed if they are not present in YAML

If a model has historical runs and its earliest YAML `validFrom` is too recent, the sync will fail.

## 7. Keep your main config references unchanged

Your main config should continue to reference provider model codes, not internal model grouping codes.

This remains correct:

```yaml
- provider: openai
  model: gpt-4o-mini-2024-07-18
```

## 8. Run migrations

Once the env vars and directories are in place, run:

```sh
npm run migrations:run
```

## 9. Sync the database from files

Start the app and run:

- `Update the database from files`

This should update:

- providers and models
- structured objects
- tools
- prompts
- tests

## 10. Verify the migration

The migration should be a no-op in terms of runnable history.

Check all of the following:

1. `Update the database from files` completes without errors.
2. `Run missing tests` finds nothing to do.
3. `Run missing evaluations` finds nothing to do.
4. `Check stats` still shows the expected active models and recent history.

If steps 2 or 3 suddenly find work that you did not expect, stop and inspect the migrated model registry before generating new data.

## 11. Common problems

### Missing provider/model directories

If `AI_TESTER_PROVIDERS_DIR` or `AI_TESTER_MODELS_DIR` is missing, the app fails at startup instead of guessing.

### Main config references a model that no longer exists

The app warns and skips that model. This behaves the same as removing it from the config.

### Cost history starts too late

The sync fails if the earliest YAML cost entry does not cover the oldest recorded run for an active model version.

### Duplicate active variants for the same provider model code

If you keep multiple YAML files for the same `provider` + `providerModelCode`, at most one may have `active: true`.

## After the migration

Once everything is working:

- keep providers in provider YAML files only
- keep one model version per model YAML file
- treat model YAML `costs` as the source of truth
- treat runtime options in model YAML as part of the model version definition
