# Migration guide: v0.14.0 to v0.15.0

This guide is for upgrading an existing project from `v0.14.0` to `v0.15.0`.

The goal is to move app-owned currency data out of ad hoc SQL and into YAML files without changing how your historical runs, costs, or stats behave:

- existing sessions and evaluations remain attached to the same logical model versions
- model costs continue to come from model YAML files
- currency exchange rates can now come from currency YAML files
- manual SQL remains available for schema work and truly custom/private tables or views

## What changed

This release adds an optional currency registry:

- currencies can now come from `AI_TESTER_CURRENCIES_DIR`
- currency exchange rates can now be synchronized from YAML files into `currencies` and `currency_rates`
- the `Update the database from files` workflow now syncs currency files before the other file-backed registries

If you do not configure `AI_TESTER_CURRENCIES_DIR`, legacy/manual DB-managed currency data still works as before.

## Before you start

1. Read the [changelog](/CHANGELOG.md) entry for `v0.15.0`.
2. Make a backup of your SQLite database file.
3. Make sure your current project is not mid-run.
4. If you already maintain exchange rates with manual SQL, locate those SQL files now so you can convert them cleanly.

If you are unsure where your DB is, check `AI_TESTER_SQLITE_DB_PATH` in your env file.

## 1. Update the package

Update the package version first:

```sh
npm update @gerukin/ai-tester
```

If you use the repository directly rather than the published package, update your checkout to the target version and run:

```sh
npm install
npm run compile
```

## 2. Decide whether to enable the currency registry now

The new currency registry is optional.

- If you want to keep your existing manual currency SQL for now, you can skip the registry setup and continue using your current DB-managed `currencies` and `currency_rates` rows.
- If you want to move exchange-rate data into YAML files, continue with the steps below.

## 3. Add the optional currency env var

Add this to your env file when you want file-backed currencies:

```env
AI_TESTER_CURRENCIES_DIR=.local/ai-evals-data/currencies
```

Use whatever path is appropriate for your project, but the directory must exist before syncing from files.

## 4. Create the currency directory

Create the directory on disk:

```sh
mkdir -p .local/ai-evals-data/currencies
```

## 5. Convert manual currency SQL into currency YAML files

Create one YAML file per currency under `AI_TESTER_CURRENCIES_DIR`.

Example:

```yaml
code: JPY
rates:
  - rateInUSD: 0.0064
    validFrom: 2025-01-19
```

Rules:

- `code` must be a 3-letter ISO 4217 currency code
- `rates` must contain at least one entry
- each `validFrom` must be unique within the file
- `rateInUSD` must be positive

If you previously inserted `USD` and `JPY` with SQL, create `USD.yaml` and `JPY.yaml` with the corresponding rate history.

## 6. Make sure referenced currencies are covered

If `AI_TESTER_CURRENCIES_DIR` is configured, every currency referenced by:

- any model YAML `costs[].currency`
- any analysis query `currency`

must have a matching currency YAML file.

If a referenced currency is missing from the currency registry, the sync fails instead of silently relying on old DB state.

## 7. Run migrations

Once your env vars and directories are in place, run:

```sh
npm run migrations:run
```

## 8. Sync the database from files

Start the app and run:

- `Update the database from files`

This now updates:

- currencies
- providers and models
- structured objects
- tools
- prompts
- tests

## 9. Verify the migration

Check all of the following:

1. `Update the database from files` completes without errors.
2. Stats still show expected costs for currencies you use.
3. `Run missing tests` finds nothing unexpected to do.
4. `Run missing evaluations` finds nothing unexpected to do.

If tests or evaluations suddenly find new work that you did not expect, stop and inspect the model registry or main config before generating more data.

## 10. Common problems

### Missing currency directory

If `AI_TESTER_CURRENCIES_DIR` is set but the directory does not exist, the sync fails immediately.

### Missing currency YAML for a referenced currency

If a model cost or analysis query references a currency with no YAML file, the sync fails and tells you which reference is missing.

### Duplicate rate dates

Each `validFrom` must be unique within a currency file. Duplicate entries cause the registry validation to fail.

### Invalid date format

If a `validFrom` value is not parseable as a date, the registry validation fails.

## After the migration

Once everything is working:

- keep app-owned exchange-rate history in currency YAML files
- keep model costs in model YAML files
- use manual SQL only for schema work and truly custom/private SQL assets
- keep using the existing migrations flow for schema changes
